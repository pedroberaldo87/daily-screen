# i18n — Multi-Language Support for Daily Screen

## Context

Daily Screen is a wall-mounted tablet app (ADHD morning assistant) with ~130 hardcoded Portuguese strings across display, admin, and login pages. Users who speak English or Spanish cannot use the app. This spec adds multi-language support with browser autodetection and manual override, using a lightweight vanilla JS approach that fits the project's zero-dependency philosophy.

## Decisions

- **Scope:** Full app (display, admin, login)
- **Config:** Single global `language` setting in SQLite
- **Detection:** `navigator.language` as fallback when no setting exists
- **Override:** Manual selector in admin settings modal
- **Languages:** pt-BR (default/fallback), en, es
- **Files:** One JSON per language (`public/lang/{lang}.json`)
- **API errors:** Stay in English (REST convention); frontend translates if needed

## Architecture

### Module: `public/i18n.js` (~60 lines)

**State:**
- `currentLang` — active language code
- `translations` — loaded strings for current language
- `fallbackTranslations` — pt-BR strings (always loaded)

**Exported functions:**

| Function | Purpose |
|----------|---------|
| `initI18n(settings?)` | Bootstrap: read language from settings or detect from browser, load JSON, apply to DOM |
| `t(key, params?)` | Return translated string with optional interpolation. Fallback chain: current lang → pt-BR → key literal |
| `applyTranslations()` | Walk DOM for `[data-i18n]`, `[data-i18n-placeholder]`, `[data-i18n-aria]` and apply text |
| `setLanguage(lang)` | Switch language at runtime (fetch JSON + apply). Used by admin selector |
| `getLang()` | Return current language code |

**Language detection mapping:**

```
navigator.language  →  JSON file
pt-BR, pt          →  pt-BR
en-US, en-GB, en   →  en
es-AR, es-ES, es   →  es
anything else       →  pt-BR
```

### HTML Markup Pattern

Static text:
```html
<span data-i18n="display.medications">Medicamentos</span>
```

Placeholders:
```html
<input data-i18n-placeholder="admin.searchCity" placeholder="Buscar cidade...">
```

Aria labels:
```html
<button data-i18n-aria="display.prevDay" aria-label="Dia anterior">◀</button>
```

Portuguese text stays in HTML as fallback if JSON fails to load.

### Interpolation

Simple `{var}` replacement in `t()`:

```javascript
t('display.progress', { done: 3, total: 5 })
// pt-BR: "3 de 5"
// en:    "3 of 5"
// es:    "3 de 5"
```

### Date/Time Localization

Each JSON includes:
- `_locale` — BCP-47 locale for `toLocaleDateString()` / `toLocaleTimeString()`
- `_dateFormat` — template for the full date display

```javascript
// pt-BR: "segunda-feira, 5 de abril"
// en:    "Monday, April 5"
// es:    "lunes, 5 de abril"
```

## Detection & Selection Flow

### Initialization (every page load)

```
Page loads → initI18n(settings)
  settings.language exists?
    YES → use it
    NO  → navigator.language → map to supported lang → fallback pt-BR
  fetch /lang/{lang}.json → apply
```

Both `display.js` and `admin.js` already fetch `/api/settings` on load. `initI18n` receives that settings object — no extra HTTP request.

### Admin Language Selector

New section in the settings modal (first section, before "Localização"):

```
┌─────────────────────────────────┐
│  Idioma                         │
│  [🇧🇷 Português ▼]              │
│    🇧🇷 Português                │
│    🇺🇸 English                  │
│    🇪🇸 Español                  │
└─────────────────────────────────┘
```

- `<select>` with language options
- On change: `PUT /api/settings { language: 'en' }` + `setLanguage('en')`
- Page re-translates without reload

## Backend Changes

### Settings whitelist (`routes/api.js`)

Add `'language'` to `ALLOWED_SETTINGS`. Validation: value must be one of `['pt-BR', 'en', 'es']`.

### Geocoding language

Currently hardcoded `language=pt`. Change to use the configured language setting for city name localization.

### Static file serving

Language JSONs served as static files from `public/lang/`. No new routes needed — Express static middleware handles it.

## Files to Create

| File | Content |
|------|---------|
| `public/i18n.js` | Translation module (~60 lines) |
| `public/lang/pt-BR.json` | Portuguese translations (complete, ~90 keys) |
| `public/lang/en.json` | English translations |
| `public/lang/es.json` | Spanish translations |

## Files to Modify

| File | Change |
|------|--------|
| `views/display.html` | Add `data-i18n` attributes to all translatable elements |
| `views/admin.html` | Add `data-i18n` attributes + language selector in settings modal |
| `views/login.html` | Add `data-i18n` attributes |
| `public/display.js` | Use `t()` for dynamic strings (greetings, progress, date format). Use `_locale` for date/time formatting |
| `public/admin.js` | Use `t()` for dynamic strings (weekdays, categories, messages). Init i18n on load. Add language selector handler |
| `public/icon-picker.js` | Use `t()` for category labels and search placeholder |
| `routes/api.js` | Add `language` to ALLOWED_SETTINGS with validation. Update geocoding language parameter |
| `server.js` | Add `/lang/*.json` to CSP if needed (should be covered by same-origin static serving) |

## Note: Icon Picker Keyword Search

`EMOJI_KEYWORDS` in `icon-picker.js` already contains both PT-BR and EN keywords for search. This is a search index, not UI text — it does NOT need per-language variants. Only the category labels and UI strings (placeholder, empty state) get translated.

## Verification

1. **Default state (no setting):** Open display on a pt-BR browser → shows Portuguese. Open on en-US browser → shows English.
2. **Admin override:** Set language to Spanish in admin → display and admin both switch to Spanish.
3. **Fallback:** Delete `es.json` and set language to `es` → falls back to pt-BR strings.
4. **Interpolation:** Check progress counter shows correct format per language ("3 de 5" vs "3 of 5").
5. **Date format:** Verify date displays correctly per locale (word order, prepositions).
6. **Page reload:** Refresh display → language persists (read from settings).
7. **Geocoding:** Search city with English selected → city names in English.
